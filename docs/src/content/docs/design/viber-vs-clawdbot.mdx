## Architectural Philosophy

Viber and Clawdbot take fundamentally different approaches to agent orchestration. Understanding these differences helps you choose the right tool for your use case.

<Aside type="tip">
  Viber adopts Clawdbot's excellent **Skill** format but rejects its centralized architecture in favor of a more flexible **Desktop Mesh**.
</Aside>

---

## The Flexibility Spectrum

| Aspect | Clawdbot | Viber |
|--------|----------|-------|
| **Topology** | Centralized Gateway | Decentralized Agent Mesh |
| **Workflows** | Rigid JSON/YAML pipelines | Fluid AI-driven Playbooks |
| **Data Flow** | Explicit stdin/stdout passing | Natural language delegation |
| **Routing** | Gateway-mediated | Direct agent-to-agent |
| **Control Planes** | Single Gateway | Multiple simultaneous adapters |
| **Failure Mode** | Single point of failure | Graceful degradation |

## Gateway vs Mesh

### Clawdbot's Centralized Gateway

Clawdbot centers around a **Gateway** process that serves as the central router:

```
┌─────────────────────────────────────────┐
│              GATEWAY                    │
│        ws://127.0.0.1:18789            │
│   • Routes messages to agents           │
│   • Manages session state               │
│   • Single point of control             │
└─────────────────────────────────────────┘
         ▲           ▲           ▲
    WhatsApp     Telegram      API
```

**Trade-offs:**
- ✓ Centralized control and monitoring
- ✗ Single point of failure
- ✗ Rigid topology

### Viber's Desktop Mesh

Viber uses a decentralized **Agent Mesh** where agents communicate directly:

```
┌─────────────────────────────────────────┐
│              AGENT MESH                 │
│   ┌───────┐   ┌───────┐   ┌───────┐    │
│   │ Coder │◄─►│Triage │◄─►│Browser│    │
│   └───────┘   └───────┘   └───────┘    │
│         Event Bus (internal)            │
└─────────────────────────────────────────┘
     ▲              ▲              ▲
   CLI          WebSocket         Web
         (Pluggable Adapters)
```

**Advantages:**
- ✓ No single point of failure
- ✓ Connect to multiple control planes simultaneously
- ✓ Lightweight adapter pattern
- ✓ Agent-to-agent communication without routing overhead

## Workflow Comparison

### Clawdbot's Lobster: Typed Workflows

Clawdbot uses **Lobster**, a typed workflow runtime with rigid pipelines:

```yaml
# Lobster: Deterministic and linear
pipeline:
  - collect:
      command: "find . -name '*.ts'"
  - process:
      stdin: "$collect.stdout"
      command: "parse-files"
  - output:
      stdin: "$process.stdout"
```

**Characteristics:**
- Explicit `stdin`/`stdout` data passing
- Deterministic execution order
- JSON/YAML configuration
- Predictable but inflexible

### Viber's Playbooks: Agentic Procedures

Viber uses **Playbooks**—Markdown-defined workflows interpreted by a Supervisor Agent:

```markdown
# Playbook: release-feature

## Goal
Prepare a feature branch for release.

## Steps
1. [Coder] Run all tests and ensure they pass.
2. [Coder] Bump the version in package.json.
3. [Git] Create a new branch named `release/v<version>`.
4. [Git] Push the branch and open a PR.
```

**Advantages:**
- Natural language instructions
- Dynamic planning and adaptation
- Agent selection based on capability
- Human-readable format

<Aside type="note">
  Playbooks are **guidelines**, not rigid scripts. The Supervisor Agent interprets them intelligently, adapting to unexpected situations.
</Aside>

## Skill System: Shared Excellence

Both frameworks use the `SKILL.md` format—a pattern Viber directly adopts from Clawdbot:

```markdown
---
name: git-commit
description: Stage changes and commit with a message.
parameters:
  message:
    type: string
    description: The commit message
    required: true
---
git add .
git commit -m "$message"
```

**Why this works:**
- Decoupled from core runtime code
- Self-documenting capabilities
- Easy to extend and share
- LLM-friendly metadata

## When to Choose What

<Tabs>
  <TabItem label="Choose Viber">
    **Viber is ideal when:**
    
    - Need multi-control-plane connectivity
    - Building products (not just automations)
    - Want agent-to-agent communication
    - Prefer TypeScript-first development
    - Need graceful failure handling
    - Want AI-driven workflow adaptation
  </TabItem>
  <TabItem label="Choose Clawdbot">
    **Clawdbot is ideal when:**
    
    - Need centralized monitoring and control
    - Building channel integrations (WhatsApp, Telegram)
    - Want deterministic, predictable pipelines
    - Prefer rigid workflow definitions
    - Single gateway topology is acceptable
  </TabItem>
</Tabs>
